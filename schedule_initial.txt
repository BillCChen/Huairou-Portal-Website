# 门户网站项目开发主计划

## 0. 总体判断

- 项目范围判断：该项目不是纯展示站，至少包含门户前台、CMS 后台、基础账号体系、注册审核、文件管理、站点配置、基础统计/日志、部署交付、培训与验收闭环。
- 参考文件支撑程度：现有文件足以支撑 V1 主体方案、模块拆解、页面清单、核心实体、基础权限、测试与验收框架；不足以直接冻结研究所/人才服务/在线服务/产学研合作的业务深度、公开字段边界、角色权限矩阵、第三方对接方式、上线口径。
- 当前计划适合推进阶段：可直接推进到“范围冻结、信息架构、原型/UI、数据库初版、接口定义、后台菜单设计、测试方案设计”；可对已明确模块进入详细设计准备；不适合直接冻结全部业务字段与权限边界。
- 本次规划采用原则：  
  已确认 = 合同附件、整合版附件、两份开发规格书一致的内容。  
  建议值 = 工程低风险默认规范，如 `/api/v1`、分页、文件大小、后台 PC 优先。  
  待确认 = 影响业务规则、展示口径、权限边界、验收口径、第三方对接的事项。  
  采用顺序 = 合同及原始附件/整合版附件优先，确认事项清单用于识别阻塞项，乙方工作版规格书用于补足工程执行细节，“虚拟生成”版仅作为演示导向参考，不覆盖合同与附件。

## 1. 项目总览

| 项目 | 结论 |
|---|---|
| 项目目标 | 建设研究院官方门户网站，承接机构形象展示、新闻发布、案例展示、业务入口引导、基础在线服务入口、后台内容运营、部署运维与验收交付 |
| 本期范围（V1） | 首页、新闻动态、成功案例、关于我们、登录/注册/审核、后台内容管理、分类标签、Banner/单页/页脚配置、文件上传、站点设置、基础统计与日志、容器化部署、测试与验收资料 |
| 预留范围（V2） | 研究所深度展示、成果孵化、共性平台、人才服务深度功能、产学研合作、在线服务流程化、全站搜索、高级筛选、业务统计、推荐系统、第三方系统深度对接、智能问答 |
| 建议技术边界 | 门户前台 Nuxt 4，CMS 后台 Vue 3 + Element Plus，后端 FastAPI，数据库 PostgreSQL，缓存 Redis，网关 Nginx，交付基线 Docker Compose；Kubernetes 仅作为甲方环境要求时的后续适配项 |
| 总体实施策略 | 先冻结范围与 IA，再定数据模型与接口，再做 CMS 驱动的前后台闭环，最后做性能、安全、部署、培训与验收；所有高风险业务点先参数化或预留，不写死 |

## 5. 七大部分详细开发计划

### 5.1 项目初始化

- 目标：冻结范围、术语、信息架构、角色边界、技术基线与交付口径，形成后续设计和实施的统一依据。
- 前置依赖：参考文件阅读完成；A 类确认项进入确认流程；合同 120 日硬约束明确。
- 具体任务：建立需求追踪矩阵；输出 V1/V2 范围表；冻结导航与页面清单；冻结角色清单与权限边界初版；冻结技术栈、环境分层、部署基线、日志与配置规范；形成设计输入清单与内容素材清单。
- 产出物：需求矩阵、范围冻结表、IA 草案、角色清单、技术方案说明、里程碑计划、确认项清单。
- 风险点：范围漂移、术语不统一、乙方工作版与合同附件口径不一致。
- 是否可直接开发：部分可直接推进。
- 可直接开发项：需求矩阵、技术基线、环境分层、页面清单、模块拆解、验收框架。
- 待确认项：导航最终结构、V1/V2 边界、上线环境要求、角色边界。

### 5.2 前端开发

- 目标：完成 SEO 友好、响应式、CMS 驱动的门户前台，覆盖 V1 必做页面，并为争议模块保留入口与扩展位。
- 前置依赖：IA 冻结；页面优先级明确；首页模块与关于我们口径明确；数据实体初版完成。
- 具体任务：建设前台路由与通用布局；实现首页聚合展示；实现新闻、案例、关于我们、登录/注册；实现错误页、无数据态、SEO 元信息、友好 URL；对争议模块采用“只读/静态/隐藏”策略；完成移动端核心页面适配。
- 产出物：前台页面原型映射表、前端组件清单、页面联调清单、SEO 配置方案、响应式验收单。
- 风险点：首页与导航改版成本高；研究所/人才/在线服务范围冲突；品牌素材迟到。
- 是否可直接开发：V1 核心页面可直接推进详细设计。
- 可直接开发项：首页、新闻、案例、关于我们、登录/注册、404/500/无数据页、页脚与站点配置接入。
- 待确认项：研究所/人才服务/在线服务/产学研合作是否主上线；双语要求；视觉素材与地图。

### 5.3 后端开发

- 目标：提供前后台共享的 RESTful API，覆盖内容管理、认证、审核、文件、配置、日志、基础统计。
- 前置依赖：数据模型初版；角色权限初版；登录注册规则与短信方案初版；接口风格统一。
- 具体任务：定义 API 分组与鉴权方案；实现认证与用户信息接口；实现内容 CRUD 与发布状态；实现文件上传/下载/引用；实现站点设置、Banner、单页、分类标签；实现审核与日志；为搜索、统计、在线服务、第三方对接预留接口边界。
- 产出物：接口清单、字段字典、错误码规范、鉴权说明、联调用例、API 文档。
- 风险点：业务规则未冻结时接口易返工；短信/邮件/第三方联调受外部依赖影响。
- 是否可直接开发：核心接口可直接推进详细设计。
- 可直接开发项：认证、用户、文章、案例、分类标签、单页、Banner、文件、站点设置、日志。
- 待确认项：注册审核节点、找回密码方式、在线服务流程、全站搜索与业务统计、外部系统对接。

### 5.4 数据库设计

- 目标：形成支持 CMS 驱动与后续扩展的数据模型，避免将栏目、角色、首页模块写死。
- 前置依赖：实体清单稳定；V1/V2 范围初步冻结；公开字段与敏感字段边界有结论。
- 具体任务：定义核心表、关联表、审核状态、发布状态、文件元数据、站点配置、日志表；建立软删除、排序、启用状态、SEO 字段、时间字段规范；对研究所、在线服务、下载资源、统计口径预留扩展字段。
- 产出物：ER 草图、字段字典、索引策略、状态枚举、迁移顺序建议、备份恢复设计。
- 风险点：研究所/人才/服务模块的业务深度不清，可能引发表结构返工；公开字段与后台字段混用会引发合规问题。
- 是否可直接开发：核心内容模型可直接推进，争议实体只宜做可扩展初版。
- 可直接开发项：users、roles、articles、categories、tags、cases、pages、banners、files、site_settings、audit_logs。
- 待确认项：permissions 粒度、研究所固定数量、服务预约/咨询/报名台账字段、业务统计口径。

### 5.5 CMS 后台

- 目标：完成运营人员可用的内容管理后台，支撑录入、审核、发布、配置、文件管理与基础审计。
- 前置依赖：权限矩阵初版；内容实体与页面结构稳定；发布流程有最小共识。
- 具体任务：设计后台菜单与角色菜单；实现内容列表、表单、富文本、批量操作、软删除/回收站；实现注册审核、角色管理、登录日志、操作日志；实现 Banner、首页统计卡片、快速入口、站点基础信息、SEO、页脚设置；根据范围决定是否加入研究所、资料下载、线索管理、活动报名。
- 产出物：后台原型映射表、菜单权限表、表单字段清单、运营手册目录、管理员验收清单。
- 风险点：审核流和角色矩阵未定；后台与前台字段映射不一致会造成脱节。
- 是否可直接开发：核心 CMS 可直接推进详细设计。
- 可直接开发项：新闻、案例、单页、Banner、文件库、用户审核、站点设置、日志查询。
- 待确认项：两级审核是否必做；研究所维护权限是否下放；线索管理是否纳入首版；下载资源是否要求登录。

### 5.6 部署与测试

- 目标：形成可部署、可回滚、可验证的交付链路，满足性能、安全、兼容与文档验收要求。
- 前置依赖：技术架构冻结；环境责任边界明确；测试指标明确；部署目标环境确定。
- 具体任务：设计测试环境与生产环境差异；制定功能、性能、安全、兼容、部署、文档测试计划；明确 HTTPS、密码加密、RBAC、备份恢复、日志留存策略；准备 Compose、Nginx、环境变量模板、初始化脚本；制定压测与安全扫描范围。
- 产出物：测试计划、测试用例目录、压测方案、安全检查单、部署手册、回滚手册、环境变量清单。
- 风险点：甲方正式域名/证书/服务器未到位；Kubernetes 与 Compose 口径不一致；短信服务无法联调。
- 是否可直接开发：测试设计和部署方案可直接推进。
- 可直接开发项：Compose 基线、Nginx 方案、测试计划、用例框架、回滚与备份策略。
- 待确认项：正式域名、证书、备案、生产网络策略、甲方是否提供服务器、是否要求 K8s。

### 5.7 验收准备

- 目标：将合同与附件中的验收条目转化为可执行的 UAT、文档、培训、签署闭环。
- 前置依赖：功能范围冻结；测试范围稳定；甲方验收组织方式明确；首批内容与素材到位。
- 具体任务：整理功能验收项、性能验收项、安全验收项、部署验收项、源码验收项、文档验收项；编制培训计划；准备 UAT 账号、场景、数据、问题单模板；明确试运行期间问题分级与关闭口径。
- 产出物：UAT 清单、验收用例、培训计划、交付物清单、签收清单、问题整改清单模板。
- 风险点：验收口径不统一；正式内容不到位；试运行问题无法按时闭环。
- 是否可直接开发：可直接推进验收设计与资料模板准备。
- 可直接开发项：验收矩阵、培训计划、交付物目录、试运行问题模板。
- 待确认项：UAT 组织方、签字权、缺陷关闭标准、正式内容供稿时间。

### 9.2 审核流建议

| 审核流 | 建议流程 | 当前状态 |
|---|---|---|
| 个人用户注册审核 | 注册提交 → 状态 `pending` → 审核通过 `active` / 驳回 `rejected` → 登录可用 | 可直接设计，字段细节待确认 |
| 内容发布审核 | 内容编辑保存草稿 → 提交待发布 → 审核员/管理员发布；若排期受限，可先收敛为管理员直发 | 建议值，需甲方确认是否必须两级 |
| 在线咨询/预约处理 | 用户提交 → 后台线索池查看 → 标记已处理/已联系/已关闭 | 仅建议值，是否纳入 V1 待确认 |
| 下载权限审核 | 公开下载直接下发；受限下载则登录后申请/审核 | 待确认，不宜默认硬做 |

## 10. 开发顺序、并行关系与里程碑

### 10.1 推荐顺序

1. 范围冻结与确认项收口  
2. IA/UI/数据模型/API 初版冻结  
3. 基础架构、认证、RBAC、CMS 骨架  
4. CMS 内容闭环与文件管理  
5. 前台 V1 页面与 CMS 联调  
6. 争议模块按“上线/静态/隐藏/预留”策略收口  
7. 测试、部署、培训、试运行、验收

### 10.2 并行与串行关系

| 任务 | 关系 | 说明 |
|---|---|---|
| IA 冻结 与 首页/导航 UI | 必须串行 | IA 未定时 UI 和路由会返工 |
| 数据模型初版 与 API 定义 | 可并行、需同步冻结 | 两者每天同步即可 |
| CMS 后台框架 与 前台公共组件 | 可并行 | 共享实体和配置约定即可 |
| 文件管理 与 富文本接入 | 必须串行到接口层，前后端实现可并行 | 先定上传/引用协议 |
| 注册审核 与 权限矩阵 | 必须串行 | 角色边界不清时无法定审核逻辑 |
| 前台研究所/人才/在线服务页面 | 只能在范围确认后启动深度实现 | 否则只做占位/隐藏 |
| 测试方案 与 开发实施 | 可并行 | 测试用例应前置，不等开发完成 |

### 10.3 建议里程碑

| 里程碑 | 时间建议 | 进入条件 | 完成条件 |
|---|---|---|---|
| M0 需求冻结 | 第 1 周 | 全部参考文件完成消化 | 范围表、确认项清单、导航草案、角色草案完成 |
| M1 设计冻结 | 第 2-3 周 | A 类问题至少得到暂定书面口径 | IA、UI 原型、ER 初版、API 清单、后台菜单清单完成 |
| M2 核心闭环可演示 | 第 4-8 周 | 数据模型/API 冻结 | 后台可登录、新闻/案例/单页可录入发布、前台首页/新闻/案例/关于我们可见 |
| M3 系统集成完成 | 第 9-12 周 | 核心闭环稳定 | 注册审核、文件库、站点设置、日志、争议模块的预留/静态/上线策略完成 |
| M4 测试部署完成 | 第 13-14 周 | 联调通过 | 功能/性能/安全测试完成，部署手册与交付物齐备 |
| M5 试运行与验收 | 第 15-16 周至合同第 120 日 | 测试环境和正式内容到位 | UAT 完成、培训完成、问题闭环、最终验收资料签署 |

## 11. 工作量评估

| 模块 | 工作量 | 评估依据 |
|---|---|---|
| 项目初始化与需求冻结 | M | 文档多、冲突点多，但实现量低 |
| 前端门户 V1 | XL | 页面数量多、SEO、响应式、CMS 驱动、争议模块多 |
| 后端 API | L | 内容模型清晰，但认证、审核、上传、日志、统计均需闭环 |
| 数据库设计 | M | 核心实体稳定，争议实体需做可扩展预留 |
| CMS 后台 | XL | 表单、富文本、列表、审核、站点配置、文件库、日志、权限均为核心能力 |
| 部署与测试 | L | 需满足 2 秒响应、100 并发、HTTPS、文档与回滚 |
| 验收准备 | M | 文档、培训、UAT、试运行、签署链路完整性要求高 |
| 研究所/人才/在线服务深度功能 | L 到 XL | 是否进入 V1 决定实际工作量；若含报名/预约/投递流程将显著抬升 |
| 第三方系统对接 | XL | 依赖外部接口、联调、认证方式，当前信息不足 |

## 12. 风险总表

| 风险 | 表现 | 影响 | 规避/缓解 |
|---|---|---|---|
| 需求不明确风险 | V1/V2 边界、研究所/人才/服务范围冲突 | 范围漂移、返工 | 第 1 周冻结范围表，未确认模块只做预留或静态页 |
| 甲方确认延迟风险 | A 类事项长期未答复 | 数据模型、权限、页面结构卡住 | 采用“暂定方案 + 可配置 + 书面待确认”推进 |
| 页面与 CMS 脱节风险 | 前台展示字段与后台表单字段不一致 | 内容无法正确维护 | 先做字段映射表，再做页面与 CMS 联调 |
| 数据结构返工风险 | 研究所、在线服务、统计口径未定 | 表结构与接口返工 | 核心表稳定，争议表做扩展字段或独立台账 |
| 权限模型返工风险 | 角色与审核流未定 | 后台菜单、接口鉴权、日志全部返工 | 先按“超管/内容编辑/审核员/注册用户”最小集合设计 |
| 部署环境不一致风险 | Compose 与 K8s、测试与生产差异 | 上线失败、运维成本增加 | 以 Compose 为交付基线，部署配置分层 |
| 验收口径不一致风险 | 页面做好但验收标准变化 | UAT 拖延、尾期返工 | 将合同条款转成验收矩阵，UAT 前书面确认 |
| 正式内容与素材迟到风险 | 新闻、图片、领导照片、联系方式未到位 | 页面无法真实验收 | 早做内容清单与责任人，支持占位与后台替换 |
| 短信/第三方联调风险 | 验证码登录、外部系统接口无可用环境 | 登录/联调延期 | 提前确定供应商或提供测试模式 |
| 合规风险 | 公开字段、隐私政策、备案信息未定 | 无法上线或上线后整改 | 对外字段与后台字段分离，法务页面和备案配置预留 |

## 13. 下一步建议

1. 先按本计划对 A 类确认项组织一次范围冻结会，产出书面确认纪要。  
2. 在确认纪要基础上，先细化“信息架构 + 页面原型 + ER 初版 + API 清单”四件套。  
3. 将研究所、人才服务、在线服务、产学研合作按“V1 上线 / V1 静态页 / V1 预留 / V2”四档逐项落表。  
4. 用本计划第 6-9 章作为后续分阶段实施输入，先进入“数据库与接口详细计划”或“前端页面详细计划”。
